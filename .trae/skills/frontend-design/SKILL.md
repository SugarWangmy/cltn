# Frontend Design

用于把"视觉/信息架构/交互"需求转成可落地的 UI 方案与实现指引（尤其适合原型与业务系统后台页面）。

## 何时调用

- 用户发参考图并说"复刻/对齐/重新设计/优化页面"
- 用户反馈"信息太挤/不清晰/层级不明显/看不懂/容易误解/交互不顺"
- 需要把复杂表格/筛选/看板重新组织
- 需要补齐状态：空态、禁用态、加载态、错误态、校验提示

## 输出目标

- **信息架构**：模块优先级、页面/弹窗结构、分区与层级
- **布局方案**：栅格、对齐、间距、可滚动区域、固定区域
- **组件策略**：哪些用卡片/表格/标签/按钮/弹窗/抽屉；统一样式与可复用模式
- **交互规则**：默认值、切换、排序、筛选、校验、回退、确认阻断
- **实现建议**：用现有技术栈与代码风格快速落地（Tailwind/Lucide/单文件原型）

## 设计原则（可执行）

### 1) 先做"层级"，再做"美观"

- 一个页面最多 1 个主动作（Primary）；其余动作降级为次要按钮或链接
- 关键数字优先：金额/次数/状态要放在用户视线第一屏
- 将"解释性信息"收纳：用提示、次级文字、折叠，而不是和关键数字抢空间

### 2) 信息密度可控

- 表格优先保证：列名清晰、对齐一致、可排序/筛选明确
- 当列过多：固定首列 + 横向滚动 + 列配置（或分组视图）
- 卡片/豆腐块：同一行高度一致，避免视觉抖动

### 3) 交互要可预期

- 默认值明确：默认时间、默认维度、默认排序
- "确认/取消"的后果一致：取消不改数据；确认才写入并刷新
- 阻断条件明确：欠款、缺字段、禁用项必须提示原因

### 4) 状态覆盖齐全

- 空态：无数据/无可选项/无待处理项
- 禁用态：不可用原因明确（含 tooltip 或辅助文案）
- 校验态：输入范围、格式错误、上限下限

## UI 检查清单（逐项对照）

- **对齐**：金额右对齐；标签/标题左对齐；同级元素基线一致
- **间距**：同类模块用统一间距（例如 12/16/20/24）；避免"局部挤、局部松"
- **颜色**：品牌色只用于主动作与关键可交互链接；避免全屏泛滥
- **表格**：表头可点击排序时必须有视觉暗示（箭头/下划线/高亮）
- **筛选**：时间筛选与数据筛选要明显区分（位置/背景/形状/字号）
- **可达性**：hover 不作为唯一信息入口；触控场景提供可点击区域

## 与本仓库原型的落地约束

- 优先复用现有模式：单文件 HTML、Tailwind 样式、Lucide 图标、state + render 全量重渲染、window.* 事件
- 不引入新 UI 库；不新增注释（除非用户要求）
- 修改时优先"最小改动实现可验证交互"：先把结构与口径做对，再优化细节

## 推荐工作流

1. 读参考图/现有页面：列出模块与信息层级（主信息/次信息/解释信息）
2. 画结构：确定"一屏优先展示"的模块与顺序
3. 定交互：默认值、切换、排序、筛选、校验、阻断逻辑
4. 落地实现：用现有组件风格拼出可交互原型
5. 走一遍验收：按"空态/边界/异常/可用性"检查清单自测
